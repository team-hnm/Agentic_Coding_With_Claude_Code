## 클로드 스킬
**스킬**은 특정 작업을 어떻게 수행할지에 대한 **지식/절차를 캡슐화**해서, 개인 또는 팀 단위로 **재사용 가능하게 보존**하는 형태를 말한다.

클로드 코드는 **슬래시`/` 명령**을 통해 사용자가 직접 명령을 선택해야 하지만 **스킬**은 프롬프트에 원하는 내용을 입력하면 클로드가 자동으로 해당하는 스킬을 찾아 사용한다.

### 스킬의 유형
| 구분 | 위치                 | 성격        | 유형    | 
|---|--------------------|-----------|-------|
|**개인**| `~/.claude/skills` | 개인 워크플로   | 실험적 기능| 
|**프로젝트**| `.claude/skills`   | 팀 표준 워크플로 | 프로젝트 특화 기능|   
|**플러그인**| `플로그인 번들`          | 마켓플레이스 배포 | 커뮤니티 공유|

### SKILL.md
**스킬**을 사용하기 위해서는 **SKILL.md** 파일이 필수이며 YAML 프론트매터와 마크다운으로 구성되어 있다. 

**SKILL.md** 파일은 200줄 이내로 유지하는 것이 좋으며, 상세한 내용은 별도의 **reference.md** 파일로 분리하기를 권장한다. 
이런 분량 제한을 두는 이유도 결국 하나의 스킬이 **단일 책임 원칙**을 따르도록 유도하기 위함이다. 
여러 작업을 한 스킬에 욱여넣으면 description도 모호해지고, 클로드가 언제 이 스킬을 써야 할지 판단하기도 어려워진다.

| 필드              | 필수 여부 | 설명                                                          |
| --------------- | ----- | ----------------------------------------------------------- |
| `name`          | 선택    | - 소문자·영숫자·하이픈만 사용<br>- 생략하면 디렉터리 이름을 스킬 이름으로 사용<br>- 최대 64자 |
| `description`   | 권장    | - 기능과 사용 트리거 설명<br>- 이 필드가 없으면 클로드가 스킬을 언제 써야 할지 판단하지 못함<br>- 최대 1,024자 |
| `allowed-tools` | 선택    | - 스킬 활성화 시 사용 가능한 도구 목록<br>- 예: `Read`, `Grep`, `Glob`      |
| `model`         | 선택    | - 스킬 실행 시 사용할 모델 지정                                         |
| `context`       | 선택    | - `fork`로 설정하면 격리된 서브 에이전트 컨텍스트에서 실행                        |
| `agent`         | 선택    | - `context: fork` 설정 시 사용할 서브 에이전트 유형 지정                     |

형식상으로는 모든 필드가 선택이지만, 클로드가 스킬을 제대로 찾아 쓰게 하려면 `description`은 반드시 채워야 하는 필드다. 그래서 실질적으로 중요한 필드는 `name`과 `description`이다.

스킬의 `name` 필드는 스킬이 수행하는 행위를 명확하게 드러낼 수 있는 동명사 형태를 권장하고, `description` 필드는 기능과 사용 트리거를 모두 포함해야 한다.

> **예시**
>
> name: generating-commit-messages
>
> description: git diff에서 명확한 커밋 메세지 생성. 커밋 메세지 작성 또는 스테이징된 변경 사항 검토 시 사용.

### 스킬의 점진적 동작 방식(= 동적 로딩 패턴)
스킬은 콘텐츠를 3단계로 나누어 단계적으로 로드하며, 각 단계는 서로 다른 콘텐츠를 담당한다.
이를 통해 필요하지 않은 콘텐츠가 컨텍스트 윈도우를 선점하는 것을 방지한다.

| 단계      | 레벨     | 로딩 시점              | 토큰 비용                | 콘텐츠                                                     |
| ------- | ------ | ------------------ | -------------------- | ------------------------------------------------------- |
| **1단계** | 메타데이터  | 시작 시 자동(시스템 프롬프트) | 스킬 당 약 100 토큰        | 프론트매터의 `name`과 `description`                            |
| **2단계** | 지침     | 스킬 트리거 시(자동으로 본문 로드)| 2,000 토큰(200줄) 이하 권장 | `SKILL.md` 본문의 지침과 가이드<br>워크플로우, 유의 사항, 가이드라인 등의 절차적 지식 |
| **3단계** | 리소스/코드 | 필요 시 개별 접근          | 사실상 무제한/가장 예측하기 어려운 변동 비용 | 지원 파일(문서, 스크립트, 템플릿 등)                                  |

3단계는 토큰 비용이 가장 예측하기 어려운데, 그 이유는 리소스 종류에 따라 비용 발생 방식이 다르기 때문이다.
- **참조 문서(reference.md)**: 클로드가 "읽어야" 정보를 얻으므로, 읽는 순간 그 분량이 그대로 컨텍스트에 로드된다.
- **실행 스크립트**: 코드 자체는 컨텍스트에 올라가지 않는다. 클로드가 Bash로 실행만 시키고, 결과(출력값)만 컨텍스트에 반영되므로 토큰 비용이 0에 가깝다.

스킬의 세 가지 콘텐츠 유형은 단순히 로딩 시점만 다른 게 아니라, 각자 맡은 역할과 특성도 다르다. 정리하면 다음과 같다.

| 콘텐츠 유형 | 역할 | 특성 |
|---|---|---|
| SKILL.md 본문 | 어떻게 판단할지 안내 | 유연함 (클로드가 해석) |
| 실행 코드 | 무엇을 정확히 할지 실행 | 결정론적 (항상 같은 결과) |
| 참조 자료 | 무엇이 사실인지 조회 | 정적 (찾아보기용) |

"실행 스크립트는 토큰 비용이 0에 가깝다"는 설명을 보면, "그럼 지침은 없애고 스크립트만 쓰면 되는 거 아닌가"로 오해될 수 있다.

---
> **질문**: 그럼 지침을 만들지 말고 실행 스크립트를 활용하라는 말이야?

**클로드의 답변**

아니다. "토큰 비용 0"이 의미하는 건 스크립트가 무조건 이득이라는 게 아니라, **비용이 발생하는 지점이 다르다**는 것이다.

- **실행 스크립트**: 코드 자체(50줄, 100줄짜리 스크립트)는 컨텍스트에 올라가지 않는다. 클로드가 Bash로 실행만 시키고, 결과(출력값)만 컨텍스트로 돌아온다. 코드 자체의 토큰 비용을 회피하는 구조다.
- **참조 문서**: 클로드가 "읽어야" 정보를 얻을 수 있으므로, 읽는 순간 문서 전체 분량이 그대로 컨텍스트에 로드된다. 내용을 다 읽어야 하니 그만큼 토큰을 소비하는 구조다.

다만 스크립트로 대체할 수 있는 건 **결정론적(deterministic) 작업**뿐이다. 매번 같은 입력에 같은 출력이 나와야 하는 작업(예: CSV 파싱, 정규식 추출, 이미지 리사이즈)은 스크립트로 고정하는 게 맞고, 토큰도 절약된다.

반면 아래는 스크립트로 대체가 안 된다.
- "이 코드 리뷰할 때 뭘 봐야 하는지" → 판단이 필요하므로 지침(SKILL.md)이 필요
- "이 API가 어떤 파라미터를 받는지" → 사실 조회이므로 참조 문서가 필요

간단한 결론: **판단이 필요 없고 매번 같은 결과가 나와야 하는 작업만 스크립트로 옮겨서 토큰 비용을 없애고, 판단이 필요한 부분(지침)과 사실 조회가 필요한 부분(참조 문서)은 여전히 문서 형태로 남겨야 한다.** 지침을 없애자는 말이 아니라, 지침/사실/실행 중 실행에 해당하는 부분만 스크립트로 옮기라는 뜻이다.
