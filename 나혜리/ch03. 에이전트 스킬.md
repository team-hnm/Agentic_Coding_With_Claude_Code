## 클로드 스킬
**스킬**은 특정 작업을 어떻게 수행할지에 대한 **지식/절차를 캡슐화**해서, 개인 또는 팀 단위로 **재사용 가능하게 보존**하는 형태를 말한다.

클로드 코드는 **슬래시`/` 명령**을 통해 사용자가 직접 명령을 선택해야 하지만 **스킬**은 프롬프트에 원하는 내용을 입력하면 클로드가 자동으로 해당하는 스킬을 찾아 사용한다.

### 스킬의 유형
| 구분 | 위치                 | 성격        | 유형    | 
|---|--------------------|-----------|-------|
|**개인**| `~/.claude/skills` | 개인 워크플로   | 실험적 기능| 
|**프로젝트**| `.claude/skills`   | 팀 표준 워크플로 | 프로젝트 특화 기능|   
|**플러그인**| `플로그인 번들`          | 마켓플레이스 배포 | 커뮤니티 공유|

### SKILL.md
**스킬**을 사용하기 위해서는 **SKILL.md** 파일이 필수이며 YAML 프론트매터와 마크다운으로 구성되어 있다.

**SKILL.md** 파일은 200줄 이내로 유지하는 것이 좋으며, 상세한 내용은 별도의 **reference.md** 파일로 분리하기를 권장한다.
이런 분량 제한을 두는 이유도 결국 하나의 스킬이 **단일 책임 원칙**을 따르도록 유도하기 위함이다.
여러 작업을 한 스킬에 욱여넣으면 description도 모호해지고, 클로드가 언제 이 스킬을 써야 할지 판단하기도 어려워진다.

| 필드              | 필수 여부 | 설명                                                          |
| --------------- | ----- | ----------------------------------------------------------- |
| `name`          | 선택    | - 소문자·영숫자·하이픈만 사용<br>- 생략하면 디렉터리 이름을 스킬 이름으로 사용<br>- 최대 64자 |
| `description`   | 권장    | - 기능과 사용 트리거 설명<br>- 이 필드가 없으면 클로드가 스킬을 언제 써야 할지 판단하지 못함<br>- 최대 1,024자 |
| `allowed-tools` | 선택    | - 스킬 활성화 시 사용 가능한 도구 목록<br>- 예: `Read`, `Grep`, `Glob`      |
| `model`         | 선택    | - 스킬 실행 시 사용할 모델 지정                                         |
| `context`       | 선택    | - `fork`로 설정하면 격리된 서브 에이전트 컨텍스트에서 실행                        |
| `agent`         | 선택    | - `context: fork` 설정 시 사용할 서브 에이전트 유형 지정                     |

형식상으로는 모든 필드가 선택이지만, 클로드가 스킬을 제대로 찾아 쓰게 하려면 `description`은 반드시 채워야 하는 필드다. 그래서 실질적으로 중요한 필드는 `name`과 `description`이다.

스킬의 `name` 필드는 스킬이 수행하는 행위를 명확하게 드러낼 수 있는 동명사 형태를 권장하고, `description` 필드는 기능과 사용 트리거를 모두 포함해야 한다.

> **예시**
>
> name: generating-commit-messages
>
> description: git diff에서 명확한 커밋 메세지 생성. 커밋 메세지 작성 또는 스테이징된 변경 사항 검토 시 사용.

## 스킬의 동작 원리

### 스킬의 점진적 동작 방식(= 동적 로딩 패턴)
스킬은 콘텐츠를 3단계로 나누어 단계적으로 로드하며, 각 단계는 서로 다른 콘텐츠를 담당한다.
이를 통해 필요하지 않은 콘텐츠가 컨텍스트 윈도우를 선점하는 것을 방지한다.

| 단계      | 레벨     | 로딩 시점              | 토큰 비용                | 콘텐츠                                                     |
| ------- | ------ | ------------------ | -------------------- | ------------------------------------------------------- |
| **1단계** | 메타데이터  | 시작 시 자동(시스템 프롬프트) | 스킬 당 약 100 토큰        | 프론트매터의 `name`과 `description`                            |
| **2단계** | 지침     | 스킬 트리거 시(자동으로 본문 로드)| 2,000 토큰(200줄) 이하 권장 | `SKILL.md` 본문의 지침과 가이드<br>워크플로우, 유의 사항, 가이드라인 등의 절차적 지식 |
| **3단계** | 리소스/코드 | 필요 시 개별 접근          | 사실상 무제한/가장 예측하기 어려운 변동 비용 | 지원 파일(문서, 스크립트, 템플릿 등)                                  |

3단계는 토큰 비용이 가장 예측하기 어려운데, 그 이유는 리소스 종류에 따라 비용 발생 방식이 다르기 때문이다.
- **참조 문서(reference.md)**: 클로드가 "읽어야" 정보를 얻으므로, 읽는 순간 그 분량이 그대로 컨텍스트에 로드된다.
- **실행 스크립트**: 코드 자체는 컨텍스트에 올라가지 않는다. 클로드가 Bash로 실행만 시키고, 결과(출력값)만 컨텍스트에 반영되므로 토큰 비용이 0에 가깝다.

스킬의 세 가지 콘텐츠 유형은 단순히 로딩 시점만 다른 게 아니라, 각자 맡은 역할과 특성도 다르다. 정리하면 다음과 같다.

| 콘텐츠 유형 | 역할 | 특성 |
|---|---|---|
| SKILL.md 본문 | 어떻게 판단할지 안내 | 유연함 (클로드가 해석) |
| 실행 코드 | 무엇을 정확히 할지 실행 | 결정론적 (항상 같은 결과) |
| 참조 자료 | 무엇이 사실인지 조회 | 정적 (찾아보기용) |

이러한 **동적 로딩 패턴**을 적용한 스킬 구조는 아래와 같다.
```text
database-operations/
├── SKILL.md                         # 1단계: 핵심 원칙(100줄 이내)
├── reference/
│   ├── queries.md                   # 2단계: 쿼리 작성 상세 가이드
│   ├── migrations.md                # 2단계: 마이그레이션 패턴
│   └── optimization.md              # 2단계: 성능 최적화 기법
├── templates/
│   ├── model.py.template            # 3단계: 모델 템플릿
│   └── migration.py.template        # 3단계: 마이그레이션 템플릿
└── scripts/
    └── generate_model.py            # 3단계: 모델 생성 스크립트
```
---
> **질문**: "실행 스크립트는 토큰 비용이 0에 가깝다"는 설명을 보면, 지침을 만들지 말고 실행 스크립트를 활용하라는 말인가?

**클로드의 답변**

아니다. "토큰 비용 0"이 의미하는 건 스크립트가 무조건 이득이라는 게 아니라, **비용이 발생하는 지점이 다르다**는 것이다.

- **실행 스크립트**: 코드 자체(50줄, 100줄짜리 스크립트)는 컨텍스트에 올라가지 않는다. 코드 자체의 토큰 비용을 회피하는 구조다.
- **참조 문서**: 내용을 다 읽어야 하니 그만큼 토큰을 소비하는 구조다.

다만 스크립트로 대체할 수 있는 건 **결정론적(deterministic) 작업**뿐이다. 매번 같은 입력에 같은 출력이 나와야 하는 작업(예: CSV 파싱, 정규식 추출, 이미지 리사이즈)은 스크립트로 고정하는 게 맞고, 토큰도 절약된다.

반면 아래는 스크립트로 대체가 안 된다.
- "이 코드 리뷰할 때 뭘 봐야 하는지" → 판단이 필요하므로 지침(SKILL.md)이 필요
- "이 API가 어떤 파라미터를 받는지" → 사실 조회이므로 참조 문서가 필요

간단한 결론: **판단이 필요 없고 매번 같은 결과가 나와야 하는 작업만 스크립트로 옮겨서 토큰 비용을 없애고, 판단이 필요한 부분(지침)과 사실 조회가 필요한 부분(참조 문서)은 여전히 문서 형태로 남겨야 한다.** 지침을 없애자는 말이 아니라, 지침/사실/실행 중 실행에 해당하는 부분만 스크립트로 옮기라는 뜻이다.

### 스킬 성능 최적화
스킬은 토큰 효율성과 응답 품질의 균형을 찾아 성능을 최적화한다.

1. 정보 밀도 향상: 동일한 내용을 더 적은 토큰으로 전달할 수 있도록 핵심 정보만 간결하게 작성.
2. 조건부 로딩: 작업 유형에 따라 필요한 정보만 로드하도록 구성
3. 캐싱 고려: 자주 사용되는 정보는 `SKILL.md`에 직접 포함하고, 드물게 사용되는 정보는 별도 파일로 분리.
---
> **질문** `SKILL.md`는 클로드 세션 시작 시 name, description을 읽어 미리 해당 스킬에 대한 정보를 클로드가 가지고 있기 때문에 캐싱이라는 말을 쓴걸까?

**클로드의 답변**
세션 시작 시 name/description이 미리 로드되는 것(1단계)과는 다른 얘기다. 여기서 "캐싱"은 **2단계(SKILL.md 본문)와 3단계(참조 파일) 사이의 경계**를 가리킨다.

- **자주 쓰는 정보를 SKILL.md 본문에 직접 포함**: 스킬이 트리거되는 순간(2단계) 자동으로 같이 로드됨. 따로 파일을 또 열어볼 필요가 없음.
- **드물게 쓰는 정보는 별도 파일로 분리**: 스킬이 트리거돼도 자동으로 안 딸려오고, 클로드가 필요하다고 판단할 때만 추가로 읽음(3단계).

즉 "미리 가까이 준비해둔다"는 느슨한 비유이며, "매번 다시 찾아가서 읽어와야 하는 것(3단계 온디맨드 조회)"과
"한 번 로드될 때 이미 같이 들어와 있는 것(2단계 본문 포함)"의 차이를 캐싱/캐시 미스에 빗댄 표현이다.

| | 관련 단계 | 캐싱 비유의 의미 |
|---|---|---|
| name/description 상시 로드 | 1단계 | 스킬 존재 여부를 판단하는 재료일 뿐, "캐싱"이 아님 |
| 자주 쓰는 정보 → SKILL.md 본문에 포함 | 2단계 | 트리거 시 자동으로 같이 로드 → "캐시 히트"에 가까움 |
| 드문 정보 → 별도 파일 분리 | 3단계 | 필요할 때만 추가 조회 → "캐시 미스, 필요 시 fetch"에 가까움 |

**참고: 진짜 "캐싱(Prompt Caching)" 기능과의 구분**

Anthropic API에는 반복되는 프롬프트 앞부분을 서버 쪽에 캐시해서 비용/속도를 줄이는 **Prompt Caching**이라는 별도의 기술 기능이 실제로 존재한다.
하지만 스킬 설계에서 말하는 "캐싱 고려"는 이 API 기능을 가리키는 게 아니라, 위에서 설명한 "본문 포함 vs 별도 파일 분리"라는 설계 관점의 비유적 표현이다.

## 스킬 사용과 개발

스킬은 여러 개를 조합하여 작업을 수행할 수 있으며 각 스킬은 필요한 시점에 순차적으로 또는 병렬로 활성화된다. 이렇게 실행되는 이유는 아래와 같다.

| | 주된 이유 | 토큰과의 관계 |
|---|---|---|
| 순차 활성화 | 작업 간 의존 관계 (B가 A의 결과 필요) | 메인 컨텍스트 절약은 부수 효과 |
| 병렬 활성화 | 독립적인 작업의 속도 향상 | 메인은 아끼지만 총 소비는 늘 수 있음 |

단, "병렬 활성화"는 각 스킬이 `context: fork`로 서브 에이전트에서 격리 실행될 때 성립하는 얘기다. `fork` 없이 메인 컨텍스트 안에서 여러 스킬 지침이 로드되는 경우는, 실제로는 병렬 실행이 아니라 여러 지침이 순차적으로 컨텍스트에 함께 존재하는 상태에 가깝다.

여러 개의 스킬을 조합할 때 유의할 점은 유사한 기능의 스킬들이 동시에 활성화될 수 있다는 점이다. 이를 대비해 각 스킬은 고유한 트리거 용어를 `description`에 지정해야 한다.
---
> **질문** 스킬 사용 이유는 컨텍스트 윈도우의 낭비를 막기 위해 즉, 토큰을 절약하기 위해 사용하는 것인가?

**클로드의 답변**

아니다. 토큰 절약은 스킬의 **부수 효과**이고, 근본 이유는 **지식/절차의 재사용과 일관성 확보**다.

스킬이 없다면 벌어지는 일을 생각해보면 이유가 명확해진다.
- 매번 같은 작업(커밋 메시지 작성, 코드 리뷰 기준 등)을 요청할 때마다 사람이 직접 기준을 다시 설명해야 한다.
- 세션마다, 사람마다 설명이 조금씩 달라지면 결과의 일관성이 떨어진다.
- 팀 단위라면 각자 다른 기준으로 같은 작업을 처리하게 된다.

스킬은 이 문제를 "한 번 정리해서 파일로 보존하고, 필요할 때마다 자동으로 불러온다"는 방식으로 해결한다.
즉 스킬의 존재 이유는 **반복 설명을 없애고 기준을 표준화하는 것**이다.

토큰 절약(3단계 점진적 로딩)은 이 목적을 달성하는 과정에서 따라오는 설계상의 이점이다. 만약 스킬이 항상 전체 내용을 통째로 로드하는 구조였어도,
"지식/절차를 재사용 가능하게 보존한다"는 본질적 가치는 그대로 유지된다. 다만 그렇게 만들면 스킬이 많아질수록 컨텍스트가 감당이 안 되므로,
그 문제를 막기 위해 점진적 로딩이라는 최적화가 덧붙은 것이다.

**정리**: 스킬 = 지식/절차 재사용(본질) + 점진적 로딩(그 본질을 지속 가능하게 만드는 최적화). 토큰 절약이 목적이었다면,
애초에 스킬이라는 지식 캡슐화 구조 자체가 필요 없었을 것이다.

### 스킬 환각 방지 지침 작성

환각이란 실제로 확인하지 않은 내용을 그럴듯하게 지어내는 것을 말한다.

**코딩 스킬에서 나타나는 대표적인 환각 두 가지**

1. 이미 존재하는 걸 **다시 만들어내는** 환각
   → 클로드가 실제 코드베이스를 보지 않고 훈련 데이터 속 일반 패턴으로 추측해서 발생
   → 방지책: 작업 전에 반드시 기존 코드베이스/파일을 먼저 검색하도록 절차를 스킬에 명시

2. 판단이 필요한 작업을 클로드가 **느낌대로** 처리하는 환각
   → "모른다"고 말하는 대신 그럴듯한 답을 내야 한다는 압박 때문에 발생
   → 방지책: 결정론적으로 처리 가능한 작업(파싱, 검증, 정렬)은 스크립트로 넘긴다.

**일반적인 환각 방지 원칙 (3가지)**

위 두 가지 증상은 아래 원칙이 지켜지지 않았을 때 나타난다.

| 원칙 | 내용 |
|---|---|
| 1. 불확실성 표현 허용 지침 | 클로드가 불확실성을 인정하고 "모르겠다"라고 대답하도록 명시적으로 허용 |
| 2. 지식 범위 제한 지침 | 클로드가 참조할 수 있는 정보의 범위를 명확히 제한 |
| 3. 출처 명시 지침 | 정보의 출처를 명시하도록 하여 사용자가 정보의 신뢰성을 직접 검증 |

- 1번 환각(다시 만들어내는 환각) ← 원칙 2 위반
- 2번 환각(느낌대로 처리하는 환각) ← 원칙 1 위반

결론적으로, 클로드에게 판단/추측을 맡기면 그럴듯하지만 틀린 답(환각)을 낼 위험이 있다.
결정론적으로 처리 가능한 작업은 스크립트로 넘겨서, 클로드가 "생성"하는 대신 스크립트가 "확인/실행"한 결과를 그대로 쓰게 만드는 것이 환각을 막는 핵심 전략이다.

### 코드 실행 통합

스킬은 `scripts/` 디렉터리를 통해 실행 가능한 코드와 통합될 수 있다. 스크립트는 클로드가 토큰을 소비하지 않고 결정론적으로 처리해야 하는 작업에 적합하다.

스크립트 작성 시 오류 사항은 클로드에게 넘기지 않고 스크립트 안에서 직접 처리한다.

스크립트 실행에 문제가 발생하면 다음 세 가지를 확인한다.

- 실행 권한
- 종속성 설치
- 경로가 유닉스 스타일 슬래시인지 여부

## 스킬 고급 사용

### 스킬의 재사용성
**재사용성이 높은 스킬의 특징**
1. 일반적인 패턴: 특정 프로젝트, 기술 스택에 종속 x
2. 충분한 트리거 조건 포함: 다양한 상황에서 적용한 description
3. 다른 스킬과의 조합이 쉬움: 명확한 체크리스트, 단계별 지침

**재사용 가능한 스킬 패턴 유형**
1. 프로젝트 특화 패턴: 특정 프로젝트의 규칙과 관행을 담은 패턴
2. 도메인 특화 패턴: 특정 기술 스택이나 프레임워크에 특화된 형태
3. 범용 유틸리티 패턴: 재사용성이 가장 높은 패턴으로 거의 모든 프로젝트에 적용 가능한 패턴

### context: fork 매커니즘
스킬은 기본적으로 대화의 컨텍스트 안에서 활성화된다. 이 말은 스킬은 메인 컨텍스트 윈도우 영역을 침범한다는 의미이다.

따라서 스킬이 행하는 작업이 대규모 코드베이스 분석처럼 많은 토큰을 소비하여 메인 컨텍스트 윈도우를 오염할 가능성이 있는 경우 메인과 분리된
서브에이전트 컨텍스트에서 실행할 수 있어야 한다.

이때 사용하는 것이 `context: fork`이다. 이는 스킬 시스템을 단순한 지식 모듈에서 독립 실행 가능한 에이전트 단위로 격상시켜
토큰 효율성, 세션 격리, 보안 제한이라는 축에서 스킬의 활용 범위를 본질적으로 확장하는 메커니즘이다.
이 기능을 통해 서브에이전트의 격리된 컨텍스트를 활용하면서 스킬 고유의 자동 발견과 동적 활성화 장점을 그대로 유지할 수 있다.

사용 방법은 전혀 어렵지 않고 단순하게 `SKILL.md`에 `context: fork` 필드를 추가하면 스킬 본문이 서브에이전트에서 구동하는 프롬프트가 된다.

#### agent 필드
`agent` 필드는 `context: fork`와 함께 설정되어 있을 경우에만 의미를 가지는 **서브에이전트 동작모드 설정**이다.

**agent 설정값**
- Explore: 코드베이스 탐색에 최적화된 읽기 전용 모드
- Plan: 컨텍스트 수집 및 실행 계획 수립 모드
- general-purpose(기본값): 범용 서브에이전트로서 별도의 제약 없이 작업 수행
