## 2-1. 기본 작업 방식

### 탐색 - 계획 - 구현 - 커밋 워크플로

사용자가 높은 수준의 목표를 제시하면 클로드 코드는 작업 과정을 체계화하여 수행한다.

#### 1. 탐색
코드베이스의 구조와 맥락을 파악하는 단계로, 전체를 분석하고 구성 요소 간의 상호작용을 파악한다.
루트부터 세부 구조로 내려가는 방식이기 때문에 특정 영역으로 좁혀가는 점진적 접근법이 효과적이다.

#### 2. 계획
구현 전략을 수립하는 단계로, 클로드 코드에서는 플랜 모드를 사용하면 쉽게 진행할 수 있다. 이 단계에서는 구현에 필요한 파일 목록, 수정 순서, 예상 영향 범위를 정리해준다.

복잡한 아키텍처 결정이나 다단계 구현이 필요할 경우 확장 사고를 활용하지만, 토큰 사용량이 크므로 주의 깊게 사용하거나 설정에서 최대 토큰 사용량을 조절해야 한다.

> **플랜 모드 사용 판단법**: 변경 대상 파일이 3개 이상이거나 구현 방향이 불확실한 경우

#### 3. 구현
계획에 따라 실제 코드를 작성하는 단계로, 필요에 따라 테스트 실행과 빌드 검증까지 처리한다.

작은 단위로 나누어 검증하면서 진행하는 점진적 접근이 중요하며, 이 방식은 오류를 최소화하는 데 도움이 된다.

#### 4. 커밋
구현된 변경 사항을 깃 저장소에 저장한다. 이 단계를 잘 활용하면 작업 단위가 완료될 때마다 커밋하여 변경 이력을 쉽게 추적할 수 있다.

> 각 단계는 순환적 구조를 기반으로 한 개별 단계이므로, 무조건 순차적으로 진행되지는 않는다.

## 탐색과 분석

이 패턴을 활용하면 필요한 파일의 경로만 컨텍스트에 적재되어 불필요한 토큰 소비를 줄일 수 있으며, 광범위한 탐색으로 인한 컨텍스트 낭비도 줄일 수 있다.

이미 프로젝트 구조를 파악하고 있다면 `@`를 사용해 탐색 범위를 좁혀 명령을 수행시킬 수 있다.

> **예시**
>
> `@src/utils/auth.js 파일의 로직을 설명해 줘`

### 플랜 모드

탐색과 분석 단계에서 의도치 않게 코드가 수정될 수 있다. 이를 예방하기 위해 존재하는 것이 **플랜 모드**다.

플랜 모드는 코드를 수정하지 않고 안전하게 개발 계획을 세울 수 있는 읽기 전용 모드로, 다음과 같은 상황에 적합하다.

1. **다단계 기능 구현**: 여러 파일에 걸친 기능 구현 시 전체 구조 설계
2. **코드 조사 단계**
3. **대화형 협의**: 반복 피드백으로 방향을 조정할 때

**플랜 모드 활성화 방법**

| 플래그 | 설명 |
|---|---|
| `shift+Tab` | 세션 중 전환 |
| `claude --permission-mode plan` | 새 세션을 플랜 모드로 시작 |
| `claude --permission-mode plan -p` | 비대화형 모드에서 플랜 모드 사용 |

## 점진적 개발 전략

점진적 개발이란 복잡한 기능을 작은 단위로 분해해 순차적으로 구현하는 접근 방식이다. 각 단계의 결과를 검증한 뒤 다음 단계로 진행할 수 있어, 에이전틱 코딩에서 유용한 전략이다.

### 작업 범위의 명확한 정의
독립적으로 테스트 가능한 단위로 분해하며, 각 단계가 완료될 때마다 검증하고 다음 단계로 진행한다. 이 단위만으로 테스트를 실행해 성공·실패를 판정할 수 있는지가 분해의 기준이 된다.

> **남은 질문**: 개념은 알겠는데 테스트를 쪼갤 수 있는지 없는지는 어떻게 판단하지?

### 피드백 루프 활용
구조화된 피드백(테스트 결과, 린터, 빌드 로그 등)을 받으면 오류를 정확히 교정할 수 있다. 이 피드백 루프가 없으면 사용자가 유일한 검증 수단이 되는데, 매번 클로드 코드에게 하나하나 명령하는 것은 비효율적이다. 반복적인 지시를 수행하는 비용이 높기 때문에, 자동화된 검증 수단을 확보해서 사용하는 것이 효율적이다.

> **남은 질문**: 어떻게 자동화된 검증 수단을 확보하지? 그 방법을 배우려고 하는건데 말이 없네..

### 컨텍스트 관리
컨텍스트 윈도우가 소진되면 클로드 코드의 응답 품질이 저하되기 때문에, 여러 방법으로 컨텍스트를 정리하거나 압축한다.

| 명령어 | 기능 |
|---|---|
| `/compact` | 대화 내용 요약 |
| `/clear` | 대화 내용 정리 및 새로 시작 |
| `/mcp` | 사용하지 않는 서버 비활성화 |
| 자동 압축 | 임계치 도달 시 자동 요약 |

## TDD 패턴

**TDD(테스트 주도 개발)**는 점진적 개발 전략을 가장 효율적으로 사용할 수 있는 방법론으로, **레드 - 그린 - 리팩터 사이클**을 사용한다.

#### 레드 단계
구현할 기능의 예상 동작을 정의하는 테스트를 먼저 작성한다. 구현된 기능이 없으므로 테스트는 반드시 실패한다.

#### 그린 단계
레드 단계에서 작성한 테스트 코드를 통과하기 위한 최소한의 코드를 작성한다. 이 단계에서 클로드 코드는 기능을 구현하고 테스트를 실행하여, 결과가 성공할 때까지 코드를 수정한다.

#### 리팩터 단계
그린 단계에서 테스트를 통과한 코드를, 기능은 유지하면서 구조를 개선하거나 중복을 제거하는 단계다.

> TDD에서는 테스트 범위의 누락 부분을 파악해 엣지 케이스로 보완하는 것이 중요하다.

## 깃 작업 트리

작업 트리는 하나의 로컬 저장소를 공유하면서 여러 개의 브랜치를 서로 다른 폴더(작업 디렉터리)에서 동시에 체크아웃하고 작업할 수 있게 해주는 기능이다.

**작업 트리 생성 명령어**
```
git worktree add <폴더 경로> -b <새로운 브랜치명> <기준 브랜치>
```

각 작업 트리에서 별도의 클로드 코드 세션을 실행하면, 독립된 작업 공간에서 병렬로 개발할 수 있다.

### `git branch` vs `git worktree`

| 구분 | 역할 |
|---|---|
| `git branch` | 하나의 저장소 안에서 작업 흐름을 나누는 커밋 포인터 |
| `git worktree` | 여러 브랜치를 각각 다른 폴더에 체크아웃해 동시에 작업할 수 있게 하는 기능 |

즉 `branch`는 작업의 버전이고, `worktree`는 그 브랜치를 펼쳐놓은 작업 폴더다.

> **남은 질문**: worktree는 branch보다 큰 개념일까?

## 2-2. 설정 관리

`settings.json` 파일을 통해 클로드 코드의 동작을 제어하는 설정을 할 수 있다. 
이 파일은 팀 규칙뿐만 아니라 개인 설정까지, 여러 수준의 설정을 관리할 수 있도록 계층적 구조를 제공한다.

### settings.json의 특징
- **형식**: JSON
- **제어 설정**: 권한 규칙, 환경 변수, 도구 설정 등
- **적용 범위**: 파일이 배치된 위치에 따라 달라짐
- **확인 및 수정 명령어**: `/config`

> 적용 범위를 다양하게 둘 수 있어, 여러 위치의 설정 파일을 계층적으로 병합해 최종 설정을 결정한다.

### 클로드 코드 설정 5단계

| 단계 | 설정 파일 위치 | 특징 |
|---|---|---|
| 1. 관리자 설정 | (조직 관리 도구) | 사용자 재정의 불가, 조직 전체의 보안 정책 강제 |
| 2. 엔터프라이즈 정책 | 시스템 디렉터리 | 사용자 재정의 불가, 인프라팀이 배포 |
| 3. 프로젝트 설정 | `.claude/settings.json` | 코드 컨벤션, 테스트 절차, 프로젝트별 도구 권한 저장 |
| 4. 프로젝트 로컬 설정 | `.claude/settings.local.json` | 팀 표준을 따르되 개인별로 조정 |
| 5. 사용자 설정 | `~/.claude/settings.json` | 모든 프로젝트에 적용. 개인 선호 모델, 전역 권한 규칙, 원격 분석 비활성화 등 |

이 5단계는 **어떤 설정 파일이 다른 파일보다 우선하는가**, 즉 파일 간 우선순위를 정하는 계층이다. 숫자가 작을수록(관리자 설정 쪽) 우선순위가 높다.

---

### 프로젝트별 설정 전략

`.claude/settings.json`의 `permissions` 객체는 `allow`, `deny`, `ask` 세 개의 배열로 도구 사용 권한을 규칙 단위로 관리한다.

| 배열          | 동작              |
|-------------|-----------------|
| `allow`     | 별도 확인 없이 자동으로 허용 |
| `deny`      | 무조건 차단          |
| `ask`       | 실행 전 사용자에게 확인 요청 |
| `workspace` | 접근 가능한 디렉터리 추가  |

**규칙 우선순위**: `deny` → `ask` → `allow` 순으로 적용된다. 어떤 도구 호출이 `deny` 규칙에 해당하면, 
같은 대상이 `allow`나 `ask`에도 걸쳐 있더라도 다른 규칙과 상관없이 차단된다.

한번 승인된 명령은 프로젝트 및 명령 당 영구적으로 저장하며, 승인 기록은 세션 종료 시 초기화 된다.

> **5단계 설정 계층과 구분할 것**
>
> - **5단계 계층**: 어떤 **설정 파일**이 다른 파일을 이기는가에 대한 우선순위 (관리자 > 엔터프라이즈 > 프로젝트 > 프로젝트 로컬 > 사용자)
> - **deny-ask-allow**: 하나의 설정 파일 **안에서**, `permissions`의 세 규칙 타입 중 무엇이 다른 규칙 타입을 이기는가에 대한 우선순위
>
> 즉 하나는 "파일 간" 우선순위, 하나는 한 파일 안의 "규칙 타입 간" 우선순위로, 서로 다른 축의 개념이다. 
> 예를 들어 프로젝트 설정 파일(`.claude/settings.json`)의 `deny` 규칙은 같은 파일 안의 `allow` 규칙보다 우선하며, 
> 이 관계는 사용자 설정이나 엔터프라이즈 정책과의 파일 간 우선순위와는 별개로 작동한다.

### 환경변수 관리법
1. 셸에 직접 export(우선순위 높음)
2. settings.json에 env 필드 정의

### 팀 표준 설정 구축
팀 협업을 위해 권한 규칙인 settings.json과 함께 CLAUDE.md, 스킬을 정의해야 한다.
- 권한 규칙(settings.json): 무엇을 허용, 차단할지 정의
- CLAUDE.md: 어떤 맥락과 규칙을 따를지 정의
- 스킬: 어떤 워크플로를 반복 실행할지 정의

### 클로드 코드의 확장 도구
클로드 코드는 확장이 가능한 플랫폼으로, 스킬, 서브에이전트, MCP 서버, 훅, 플러그인이라는 독립적인 확장 방식을 제공한다. 
각 도구는 서로 다른 설계 철학과 실행 컨텍스트를 가지고 있으며 해결하고자 하는 문제의 성격도 다르다.

| 항목        | 스킬                     | 서브에이전트             | MCP 서버             | 훅                 | 플러그인          |
|-----------| ---------------------- | ------------------ | ------------------ | ----------------- | ------------- |
| **핵심 목적** | 지식/지침 확장               | 작업 위임 및 격리         | 외부 도구/데이터 연결       | 생애 주기 자동화         | 확장 패키징/배포     |
| **실행 주체** | 클로드(AI 추론)             | 독립 AI 인스턴스         | 외부 서버 프로세스         | 셸 명령(결정론적)        | 포함된 구성 요소에 의존 |
| **활성화 방식** | 자율적(설명 매칭)             | 자동 위임 또는 명시적 호출    | 도구 호출 시 자동         | 이벤트 트리거(매처)       | 설치 시 자동       |
| **컨텍스트**  | 메인 대화 공유               | 독립 컨텍스트 윈도우        | 메인 대화에서 호출         | 대화 외부(셸)          | 해당 없음         |
| **도구 접근** | `allowed-tools`로 제한 가능 | `tools` 필드로 명시적 제한 | 서버별 도구 노출          | 도구 실행 전후 개입       | 포함된 구성 요소 상속  |
| **지속성**   | 파일 기반(영구)              | 세션 내(resumable)    | 구성 기반(영구)          | 구성 기반(영구)         | 설치 기반(영구)     |
| **공유 범위** | 개인/프로젝트                | 프로젝트/사용자           | 로컬/프로젝트/사용자        | 설정 파일 수준          | 마켓플레이스        |
| **대표 사용 사례** | 커밋 메시지 규칙, 코딩 가이드      | 코드 리뷰, 디버깅, 보안 분석  | 깃허브 연동, DB 쿼리, 센트리 | 자동 포매팅, 파일 보호, 로깅 | 팀 도구 세트 배포    |

#### 스킬
클로드 코드에서 가장 가볍고 선언적인 확장 방식으로 **무엇을 알고 있는가**에 대한 기반으로 **지식을 확장**하며, 사용자 요청의 맥락을 분석하여 활성화 여부를 결정한다.

#### 서브에이전트
스킬보다 한 단계 높은 수준의 확장 방식으로 스킬과 다르게 **무엇을 할 수 있는가**에 대한 기반으로 **능력을 확장**한다.
독립적인 컨텍스트 윈도우, 커스텀 시스템 프롬프트 그리고 명시적으로 할당된 도구 세트를 갖춘 **특화된 AI 어시스턴드**로 동작한다.

서브에이전트가 독립된 단위로 기능할 수 있는 이유는 **컨텍스트 격리**에 있다.

> **컨텍스트 격리?**
> 
> 작업 과정에서 발생하는 컨텍스트 소비를 메인 세션으로부터 차단하고, 결과만 전달받는 구조

메인 대화의 컨텍스트를 오염하지 않도록 별도의 컨텍스트에서 작업을 진행하며, 복잡한 코드 리뷰나 디버깅을 주로 위임 받는다.

#### MCP 서버
클로드 코드를 **외부 세계와 연결하는 통로**로 외부 도구, 데이터베이스, API에 대한 접근을 제공한다. 
MCP 서버는 HTTP, stdio 전송 방식을 주로 사용한다.

MCP 도구에 대한 권한은 서버 수준과, 도구 수준으로 구분되며 세분화된 권한 체계를 통해 MCP 서버의 기능을 필요한 만큼만 활성화할 수 있다.

#### 훅
LLM의 판단을 우회하여 결정론적으로 실행되는 셸 명령으로, 클로드 코드의 생애 주기 이벤트에 바인딩하여 매처(matcher)패턴을 통해 특정 도구 또는 상황에만 반응하도록 설정할 수 있다.

훅은 병렬로 실행되며 동일한 훅 명령은 자동으로 중복 제거된다.

#### 플러그인
스킬, 서브에이전트, MCP 서버, 훅을 하나의 배포 단위로 묶는 컨테이너이다. 플러그인은 확장 수단 자체가 아니라 여러 확장 수단의 조합을 재사용 가능한 패지키로 만드는 메타 확장이다.

## 2-3. IAM 권한 관리
누가 무엇을 할 수 있는가를 정의하는 보안 프레임워크로 클로드 코드에서는 AI 에이전트가 파일 시스템, 셸 명령, 외부 서비스에 접근하는 방식을 제어하며 인증과 권한을 다룬다.

### IAM 인증과 권한
인증은 클로드 코드 설정을 위해 필요한 앤트로픽 모델 접근 방식을 결정하며 클로드 API, 아마존 베드락, 구글 버텍스 AI 인증 옵션이 있다. 

#### 클로드 API 인증
클로드 콘솔을 통해 직접 앤트로픽의 API에 접근하는 방식으로 사용자에게 **클로드 코드 역할**과 **Developer 역할**, 2가지의 역할을 할당할 수 있다.
- 클로드 코드 역할: 클로드 코드용 API 키만 생성할 수 잇는 제한된 권한 부여.
- Developer 역할: 모든 종류의 API 키를 생성할 수 있는 더 넓은 권한 부여.

보안원칙에 따라 사용자에게 필요한 최소한의 권한만 부여하는 것이 바람직하다.

#### 아마존 베드락 인증
AWS 인프라를 사용하고 있다면 해당 인증을 사용하며, 기존 AWS IAM 정책과 통합된다.

#### 구글 버텍스 AI 인증
GCP 자격 증명을 통해 접근할 수 있다.

### 클로드 코드의 권한 모드
| 모드                  | 설명                                         |
|---------------------|--------------------------------------------|
| `default`           | 새로운 프로젝트나 익숙치 않은 코드 작업 시 적합                |
| `acceptEdits`       | 파일 편집 권한 자동 수락, 리팩토링 작업 시 적합               |
