# 1-1 클로드 코드와 에이전틱 코딩
## AI들의 특징
- OpenAI(chatGPT): 텍스트, 영상, 이미지 생성 등을 아우르는 범용 AI
- 구글(Gemini): 검색과 클라우드 서비스 통합 주력
- 앤트로픽(Claude): 에이전틱 AI 제품화

## 클로드의 강점
1. 코드 생성의 정확도
2. 긴 문맥 처리 능력

이러한 강점을 개발자의 로컬 환경으로 이식 -> **클로드 코드**

- 클로드 코드: 소프트웨어 개발에 특화된 에이전틱 코딩 도구
- 클로드 코드 CLI: 클로드 코드를 터미널에서 실행하는 **인터페이스**

## 에이전틱 코딩
- 기존 AI 코딩 도구: 코드를 예측하는 보조 도구
- **에이전틱 코딩**: AI가 파일 시스템 탐색, 코드 수정, 테스트 실행까지 **자율**로 수행하는 접근 방식.

사람은 목표와 제약 조건만 정의하고 에이전트는 허용된 모든 도구를 직접 호출하며 작업을 완료한다. 

결국 일반 개발자들이 하던 일은 에이전트가 하고, 상급자, 시니어, 리더들이 하던 목표 정의와 최종 검수로 역할이 변화.

#### 에이전틱 코딩의 작업 과정
각 단계는 내장 도구로 자동 실행하며, 개발자가 필요한 단계에서 개입해 방향 조정.
1. 개발자가 목표를 제시
2. 하위 작업으로 분해
3. 실행 계획 수립
4. 완수

=> 정확하게 **탐색-계획-구현-커밋 워크플로** 과정으로 정의된다.

### 에이전틱 코딩 도구
구현 방식에 따라 IDE형 도구와 CLI기반 도구로 나뉜다.
- IDE형 도구: 커서
- CLI기반 도구: 클로드 코드

IDE형 도구는 에디터에 에이전트 모드를 통합하여 익숙한 에디터의 GUI 환경에서 에이전틱 기능을 사용한다. 

반면 CLI기반 도구는 에디터 없이 터미널 자체에서 작동하여 특정 에디터에 종속되지 않는다. MCP 서버를 연결하면 외부 서비스까지 접근 범위가 확장된다.

> **CLI 기반 에이적틴 도구가 외부 서비스에 접근할 수 있는 이유.**
> 
> 로컬 터미널에서 실행되기 때문에 로컬 환경의 파일, 명령어, 인증정보, 네트워크를 활용할 수 있고, 그 결과 외부 서비스까지 작업할 수 있다. 
> 
> 다만 가능한 작업 범위는 터미널/OS/외부 서비스/도구 설정의 권한에 의해 결정된다.

CLI기반 에이전틱 코딩 도구는 터미널에서 작동하기 때문에 대량의 로그를 분석하는 데에도 엄청난 효율성을 보인다.

### 에이전틱 루프
클로드 코드가 작동하는 루프 방식을 에이전틱 루프라고 한다.

![loop.png](image/loop.png)

질의응답식의 일회성 대화와 달리 컨텍스트 수집, 작업 수행, 결과 확인 후 문제 발견 시 처음으로 돌아가 작업을 다시 수행한다.

#### 1단계. 컨텍스트 수집
프롬프트를 받아 즉시 작업을 수행하는 것, 즉 코드를 작성하는 것이 아니라 프로젝트의 구조를 파악하며 프롬프트와 관련 있는 정보를 수집한다.

#### 2단계. 작업 수행
1단계에서 얻은 컨텍스트를 기반으로 실행 계획을 세우고 필요한 도구를 호출한다.

#### 3단계. 결과 확인
클로드 코드 자신이 수행한 작업을 스스로 검증한다.

검증은 3가지 종류로 나눌 수 있다.
1. 규칙 기반 피드백: 명확한 규칙으로 검증(like 린팅)
2. 시각적 피드백: 스크린샷, 렌더링 결과로 시각적 출력을 평가
3. LLM기반 피드백: 다른 언어 모델에 의해 평가 받는 것.

검증으로 문제가 발생하면 다시 1단계로 돌아가 수정을 반복한다.

> **각 단계는 고정된 순차적 실행이 아니라 상황에 따라 유동적으로 진행**한다.

## 컨텍스트 윈도우
컨텍스트 윈도우는 대화의 진행에 따라 사용자의 프롬프트, 클로드의 응답, 도구 호출과 결과가 순서대로 저장된다.

클로드가 지원하는 컨텍스트 윈도우는 20만 토큰으로 토큰 수가 증가하면 정보 회상의 정확도가 떨어지고 장거리 추론 능력이 약화된다. 

(따라서 필요한 부분만 지정하여 작업할 수 있도록 프롬프트를 작성하는 것이 중요!)

컨텍스트 윈도우의 크기는 작업의 복잡성과 밀접. 컨텍스트를 무분별하게 채울 경우 응답 품질이 저하될 수 잇다. 어떤 정보를 컨텍스트레 포함할지 항상 의식적으로 관리해야 함.

### 컨텍스트 윈도우 관리 전략
1. 컴팩션: 컨텍스트 윈도우 사용량이 98%에 도달 -> 이전 대화 내용 자동 요약 => 긴 작업에도 에이전트 중단 x
2. /clear 명령: 대화 기록을 완전히 초기화, 서로 관련 없는 작업을 연속으로 수행할 때 유용.
3. 점진적 공개: 처음부터 모든 정보 로드 x, 필요할 때 도구로 데이터를 동적 로드. 
4. 서브에이전트 활용: 서브에이전트는 메인 에이전트와 별도로 독립된 컨텍스트 윈도우 사용. -> 여러개의 서브에이전트가 각각 독립적으로 작업 후 메인에서 요약 및 종합.
5. 구조화된 메모 작성: 컨텍스트 외부에서 에이전트가 md파일로 체크리스트를 작성하여 필요시 로드.

### 클로드 코드의 4대 핵심 기능
1. 자연어 기반의 기능 구축
2. 심층 디버깅 및 문제 해결
3. 코드베이스 탐색과 분석
4. 반복 태스크 자동화

여기까지 학습하고 든 생각. 결국 프로젝트 구조든 외부 서비스든 전체적으로 어느정도의 체계가 잘 잡혀있어야 에이전트 사용도 유용할 것 같음.

## 단어 학습
- 린트(Lint)/린터(Linter): 소프트웨어 정적 분석 도구로 프로그램의 오류, 버그, 스타일 문제 등을 찾아내 표시해주는 자동화 도구.
- 린팅(Linting): 린트를 사용해 코드의 품질과 규약을 검증하고 수행하는 일련의 과정.
- 에이전틱 AI: 스스로 문제를 분석하고 도구를 활용하여 자율적으로 작업을 수행하는 AI시스템, 작업을 제안하고 실행까지 완수하는 것이 특징.
- MCP(Model Context Protocol): AI 모델이 외부 데이터 소스 및 도구와 **상호작용**할 수 있도록 표준화된 프로토콜, **장치들을 연결해주는 USB 역할**
- 컨텍스트 윈도우: 한 번의 추론에서 참조할 수 있는 토큰의 총량.
