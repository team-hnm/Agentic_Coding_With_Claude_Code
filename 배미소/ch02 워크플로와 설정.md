## 내용 정리

### 점진적 개발 전략

- 복잡한 기능을 한 번에 구현하지 않고, 작은 단위로 나누어 하나씩 구현하고 검증하는 방식
- 점진적 개발은 크게 세 가지 요소로 구성된다.
    - **작업 분해**
        - 복잡한 기능을 각각 독립적으로 검증할 수 있는 작은 작업으로 나눈다.
        - 한 단계를 구현한 뒤 정상 동작하는지 확인하고 다음 단계로 넘어간다.
        - 분해 기준: **해당 작업만으로 성공과 실패를 판단할 수 있는가**
    - **피드백 루프**
        - 클로드 코드는 테스트 결과, 린트 결과, 빌드 로그처럼 명확한 결과를 받으면 이를 바탕으로 문제를 수정하고 다음 작업을 판단할 수 있다.
        - UI 작업에서는 실제 화면이 의도대로 구현되었는지 확인할 수 있는 시각적 피드백도 유용하다.

          > MCP를 통해 브라우저 외부 도구 등을 연결하여 UI 결과를 확인하는 피드백 루프를 구성할 수 있다.
        - 검증 수단이 없다면 결국 사람이 직접 결과를 확인해야 한다.
        - 따라서 클로드 코드에게 작업을 맡기기 전에 테스트, 린트, 빌드처럼 **결과를 자동으로 확인할 수 있는 환경을 만들어두는 것이 중요하다.**
    - **컨텍스트 관리**
        - 개발 세션이 길어지면 대화와 작업 정보가 계속 쌓이면서 컨텍스트 관리가 중요해진다.
        - `/compact`: 지금까지의 대화를 요약해 필요한 정보를 남기고 컨텍스트를 정리한다.
        - `/clear`: 기존 대화 컨텍스트를 초기화하고 새로 시작한다.
        - `/mcp`: 현재 작업에 필요하지 않은 MCP 서버를 비활성화해 불필요한 컨텍스트 사용을 줄인다.
        - 클로드 코드는 컨텍스트 한계에 가까워지면 기본적으로 자동 압축을 수행하며, `/compact`를 사용해 직접 압축할 수도 있다.
- **TDD 기본 패턴**
    - 점진적 개발과 잘 맞는 개발 방식 중 하나이다.
    - **실패하는 테스트 작성 → 테스트를 통과하는 최소한의 코드 구현 → 코드 구조 개선**의 `Red → Green → Refactor` 사이클을 반복한다.
    - 각 단계에서 테스트 결과가 명확한 성공/실패 신호가 되기 때문에, 클로드 코드 역시 결과를 바탕으로 다음 작업을 판단하기 쉽다.

### 깃 통합 워크플로

- 여러 기능을 동시에 개발할 때는 `Git worktree`를 활용할 수 있다.
- 하나의 Git 저장소에서 여러 작업 디렉터리를 만들고, 각 디렉터리에서 서로 다른 브랜치를 체크아웃할 수 있다.
- 새 작업트리를 생성하고, 각 작업트리에서 별도의 클로드 코드 세션을 실행하면 독립된 작업 공간에서 병렬로 개발할 수 있다.

> **Git worktree**
>
> 일반적인 브랜치는 **Git상에서 코드의 버전을 분리하는 기능**이고, worktree는 **실제 작업 디렉터리까지 여러 개로 분리하는 기능**이다.
>

> **궁금한 점: feature 브랜치를 따서 각각 별도의 클로드 코드 세션을 실행하는 것도 병렬 개발 아닌가?**
>
> 같은 프로젝트 폴더에서는 한 시점에 하나의 브랜치만 체크아웃할 수 있기 때문에, 세션을 두 개 실행하더라도 결국 같은 파일을 바라보게 된다. 한쪽 세션이 브랜치를 바꾸거나 파일을 수정하면 다른 세션에도 영향을 줄 수 있다.
>
> 반면 worktree는 **브랜치 + 실제 작업 폴더를 함께 분리**하기 때문에 각 클로드 코드 세션이 서로 다른 파일을 수정할 수 있다. 그래서 AI 에이전트 여러 개를 동시에 실행하는 병렬 작업에서는 worktree가 더 안전한 방식이다.
>

### 팀 표준 설정 구축

- **권한 규칙 (`settings.json`)**
    - 클로드 코드가 어떤 파일이나 명령어에 접근할 수 있는지 허용·차단 범위를 설정한다.
    - 팀에서 공통으로 지켜야 할 보안 규칙이나 도구 사용 범위를 설정할 수 있다.
    - 특정 Bash 명령어나 민감한 파일에 대한 접근을 차단할 수 있다.
- **`CLAUDE.md`**
    - 클로드 코드가 프로젝트에서 따라야 할 맥락과 작업 규칙을 정의한다.
    - 빌드, 테스트, 린트, 배포처럼 자주 사용하는 명령어나 팀의 공통 작업 방식을 기록할 수 있다.
    - 프로젝트 범위의 `CLAUDE.md`는 깃 저장소에 커밋해 팀이 함께 사용할 수 있다.
    - `CLAUDE.local.md` 는 개인용 지침으로 클로드 코드가 gitignore 대상으로 자동 설정한다.
- **스킬**
    - 반복적으로 수행하는 작업 방법이나 절차를 정의해 재사용하는 기능
    - `PR 리뷰 절차`, `폼 리팩터링 절차`, `QA 체크리스트`처럼 반복되는 작업 흐름을 스킬로 만들어 필요할 때 활용할 수 있다.

### 민감한 파일 보호

- `settings.json`의 권한 설정을 통해 클로드 코드가 접근하면 안 되는 파일이나 명령을 제한할 수 있다.
- 환경변수 파일, 시크릿 디렉터리, 자격 증명 파일처럼 AI가 읽거나 수정하면 안 되는 대상은 미리 접근을 차단해두는 것이 좋다.

### 클로드 코드의 다섯 가지 확장 도구

**스킬**

- 반복적인 작업 방법을 클로드 코드가 재사용할 수 있도록 정의하는 가장 가볍고 선언적인 확장 방식
- `SKILL.md` 를 중심으로 구성하고 필요에 따라 스크립트, 템플릿, 참고 문서 등을 함께 둘 수 있다.
- 클로드가 프론트매터에 기술된 `name`과 `description` 필드를 기반으로 사용자 요청의 맥락을 분석하여 활성화 여부를 결정한다.

**서브에이전트**

- 메인 클로드가 모든 작업을 직접 처리하는 대신, 특정 유형의 작업을 서브에이전트에게 맡길 수 있다.
- 각 서브에이전트는 **독립적인 컨텍스트 윈도우, 커스텀 시스템 프롬프트, 명시적으로 할당된 도구 세트를 갖춘 특화된 AI 어시스턴트**로 동작한다.

  >
  >
  >
  > 예를 들어 코드 리뷰 에이전트를 만든다면,
  > 메인 대화와 분리된 독립적인 컨텍스트에서 작업하도록 하고,
  > **"코드를 수정하지 않고 문제점만 분석한다"**는 역할을 프롬프트로 정의할 수 있다.
  > 또한 파일 읽기와 검색만 가능하도록 사용할 수 있는 도구를 제한할 수 있다.
  > ```bash
    > ---
    > name: code-reviewer
    > description: 코드를 검토하고 문제점을 분석하는 코드 리뷰 에이전트
    > tools: Read, Grep, Glob
    > ---
    > 
    > 당신은 코드 리뷰 전문 에이전트입니다.
    > - 코드를 직접 수정하지 않습니다.
    > - 버그, 보안, 가독성 측면에서 문제점을 분석합니다.
    > - 발견한 문제와 개선 방향을 설명합니다.
    >
    > ... 
    > ```
- 별도의 컨텍스트에서 작업을 진행하므로 조사나 분석 과정에서 발생한 대화가 메인 대화 컨텍스트에 그대로 쌓이지 않으며 실행 결과만 요약되어 반환한다.
- 필요한 경우 클로드가 작업을 서브에이전트에게 위임하거나 사용자가 직접 호출할 수 있다.

**MCP 서버**

- 클로드 코드가 외부 서비스나 데이터에 접근할 수 있도록 연결해주는 방식
- 외부 API, 데이터베이스, 개발 도구 등을 MCP를 통해 연결할 수 있다.
- MCP 서버는 HTTP, stdio 전송 방식을 주로 사용한다.
  > **`stdio` 방식이란? (Standard I/O)**
  > 클로드코드가 MCP를 로컬에서 실행할 때 쓰는 방식.
  > 프로세스의 표준 입력(stdin)과 표준 출력(stdout)을 통신 통로로 사용한다.
  > 로컬 파일, CLI, 개발 도구 등 내 컴퓨터의 환경과 연결되는 MCP 서버를 사용할 때 활용한다.

**훅**

- 특정 상황이 발생했을 때 미리 정해둔 동작을 자동으로 실행하는 기능
- AI가 매번 실행 여부를 판단하는 것이 아니라, 설정한 조건에 해당하면 정해진 동작이 실행된다.

**플러그인**

- 스킬, 서브에이전트, MCP 서버, 훅 등의 확장 기능을 하나의 패키지로 묶어 배포하고 재사용하는 방식
- `.claude-plugin/plugin.json` 매니페스트를 통해 메타데이터를 정의한다.
- 마켓플레이스 시스템을 통해 배포 및 설치할 수 있다.

### IAM 개념

- IAM(Identity and Access Management) 누가 어떤 리소스에 접근해 무엇을 할 수 있는지를 정의하는 보안 프레임워크이다.
- 클로드코드 환경에서는 AI 에이전트가 파일 시스템, 셸 명령어, 외부 서비스에 접근하는 방식을 제어한다.
- Anthropic 계정뿐 아니라 Amazon Bedrock, Google Vertex AI 등의 인증 방식도 지원한다.
- 클로드 API 인증
    - 클로드 콘솔을 통해 앤트로픽 API에 접근하는 방식
- 아마존 베드락 인증
    - AWS의 베드락 서비스를 통해 클로드 모델을 사용하는 방식
    - AWS의 인증 정보와 IAM 권한 체계를 사용한다.
    - `CLAIDE_CODE_USE_BEDROCK` 환경 변수를 설정하여 활성화한다.
- 구글 버텍스 AI 인증
    - GCP 자격 증명을 통해 클로드 모델을 사용하는 방식
    - GCP의 인증 정보와 권한 체계를 사용한다.
    - `CLAIDE_CODE_USE_VERTEX` 환경 변수를 설정하여 활성화한다.
