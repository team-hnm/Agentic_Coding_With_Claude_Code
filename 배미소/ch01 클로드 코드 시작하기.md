# CH 01. 클로드 코드 시작하기

## 학습 목표

1. 클로드 코드가 무엇인지 이해
2. 에이전틱 루프(클로드 코드가 작업을 수행하는 방식) 이해
3. 컨텍스트 윈도우 관리 전략 이해

## 내용 정리

### 1. 에이전틱 AI와 MCP

- **에이전틱 AI**는 ****사용자의 요청에 답변만 하는 것이 아니라, **문제를 분석하고 필요한 작업을 스스로 계획하고 수행하는 AI 시스템**이다.
- 에이전틱 AI가 실제 작업을 수행하려면 코드, 문서, 데이터베이스, 외부 API 같은 도구와 연결되어야 한다.
- **MCP**는 **AI 모델이 외부 데이터 소스 및 도구와 상호작용할 수 있도록 하는 표준화된 프로토콜**이다.

> **정리**
>

**MCP는 에이전틱 AI가 외부 도구를 활용해 실제 작업을 수행할 수 있도록 돕는 기반**이 된다.

---

### 2. 에이전틱 코딩

- **에이전틱 코딩**은 **AI가 파일 시스템 탐색, 코드 수정, 테스트 실행까지 자율적으로 수행하는 개발 방식**이다.
- 개발자가 높은 수준의 목표를 제시하면, AI는 목표를 하위 작업으로 나누고 필요한 파일을 탐색하고 구현과 검증을 수행한다.
- 일반적으로 **`탐색 - 계획 - 구현 - 커밋 워크플로`** 흐름으로 작업이 이어진다.

> **느낀점**
>

개발자가 좋은 목표와 제약조건을 정의하고, 작업 결과를 검토하는 능력이 중요하다는 생각이 다시 한 번 들었다.

특히 작업 결과가 괜찮은 변경인지 판단하는 것은 우리의 몫이기에,, 기본기 공부와 코드의 의도를 이해하는 태도를 게을리하지 않아야겠다고 느꼈다.

> **이야기해보고 싶은 부분**
>
- 개발자가 “높은 수준의 목표를 제시한다”는 것은 실무에서 어느 정도의 구체성을 의미하는지 ??

---

### 3. 에이전틱 루프

- 클로드 코드는 한 번 답하고 끝나는 방식이 아닌, 반복적인 루프를 통해 작업을 진행한다. 이를 **에이전틱 루프**라고 한다.
- **에이전틱 루프의 기본 흐름**
    1. **컨텍스트 수집**
        - 프로젝트 구조를 탐색한다.
        - 관련 파일을 검색한다.
        - 기존 코드 스타일과 구현 방식을 파악한다.
    2. **작업 수행**
        - 수집한 정보를 바탕으로 계획을 세운다.
        - 파일을 수정하거나 새 코드를 작성한다.
        - 필요한 경우 셸 명령어, 검색, 웹 접근 같은 도구를 호출한다.
        - 보조도구를 사용할 수도 있다.
            - Agent: 서브에이전트 생성
            - TodoWrite: 작업 목록 관리
    3. **결과 확인**
        - 테스트, 린터, 타입 체크 등을 실행해 결과를 검증(규칙 기반 피드백)한다.
        - 문제가 있으면 다시 원인을 탐색하고 수정한다.
        - 필요에 따라 스크린샷이나 렌더링 결과를 확인(시각적 피드백)하거나, LLM 기반 평가를 활용할 수도 있다.
- 이 루프는 항상 고정된 순서로만 움직이지 않고, 작업 상황에 따라 유동적으로 결합된다.
- 탐색과 계획 단계를 건너뛰면 클로드 코드가 충분한 컨텍스트 없이 바로 코딩을 시작할 수 있다.
- 따라서 **작업 초반에는 관련 구조를 먼저 파악하도록 유도**하는 것이 중요하다.
- 서브에이전트를 활용하면 별도의 컨텍스트에서 탐색을 수행하고 결과를 요약해 메인 에이전트에 전달할 수 있어, 메인 컨텍스트를 효율적으로 관리하는 데 도움이 된다.

> **느낀점**
>

이전에는 토큰 절약을 위해 관련 경로를 명시해준다고 생각했는데, 해당 섹션을 읽고 난 뒤 작업 범위와 컨텍스트를 좁혀주는 행동이기도 했다는 생각이 들었다.

또, 서브에이전트는 메인 에이전트가 필요하다고 판단할 때 자동으로 호출되는 것이라고 생각했는데, 개발자가 명시적으로 요청할 수도 있고, 커스텀 서브에이전트를 정의해 특정 작업에 반복적으로 활용할 수도 있다고 한다.

> **이야기해보고 싶은 부분**
>
- 서브에이전트는 어떤 상황에서 명시적으로 사용하는 것이 가장 효과적일까?

### 4. TDD와 에이전틱 루프

- 책에서는 **에이전틱 루프가 TDD 흐름과도 잘 맞는다**고 설명한다.
- TDD는 보통 다음 순서로 진행된다.
    1. 실패하는 테스트를 먼저 작성한다.
    2. 테스트를 통과하도록 최소한의 코드를 구현한다.
    3. 테스트가 통과하면 리팩터링한다.
- 클로드 코드도 이와 비슷하게 동작할 수 있다.
    - 먼저 기대 동작을 테스트로 정의한다.
    - 실패하는 테스트를 확인한다.
    - 테스트가 통과할 때까지 구현과 검증을 반복한다.
- 이 방식이 좋은 이유는 AI에게 **무엇이 정답인지 더 명확하게 알려줄 수 있기 때문**이라고 한다.
- 요구사항이 말로만 있을 때보다 테스트가 있으면 결과를 검증하기 쉬워진다.

> **느낀점**
>

TDD 흐름이 에이전틱 루프와 잘 맞는다는 점이 흥미롭다.. 문장 자체는 이해가 됐지만, 프론트를 개발하는 입장에서는 어떻게 모든 작업에 실패 테스트를 먼저 작성하지?라는 생각이 든다. 보여지는 영역 외에 유효성 검사, 데이터 가공, 권한 분기 등 결과가 명확한 부분에서는 기준이 될 수 있겠다는 생각이 든다.

> **생각해볼 것**
>
- 프론트엔드 작업에서 테스트로 확인하기 어려운 부분은 어떤 기준으로 AI에게 검증을 요청하면 좋을까?

### 5. 터미널에서 시작하는 AI 코딩

- 클로드 코드는 터미널에서 직접 실행할 수 있다.
- 터미널 기반 방식은 **모듈성**이라는 **유닉스 철학**을 따른다.
- **유닉스 철학의 핵심**
    - 한 가지 일을 잘하는 프로그램을 만들 것
    - 다른 프로그램과 함께 동작할 수 있는 프로그램을 만들 것
    - 텍스트 스트림을 처리할 수 있는 프로그램을 만들 것
- **텍스트 스트림**은 명령어 사이에서 입력과 출력으로 주고받는 텍스트 데이터다.
- 파이프(`|`)를 사용하면 앞 명령어의 출력 스트림을 다음 명령어의 입력 스트림으로 전달할 수 있다.

    ```
    cat security_logs*.log | claude -p "이 로그들을 분석해서 해킹 시도 패턴을 찾아줘"
    ```

- 이 구조 덕분에 클로드 코드는 기존 셸 명령어, 로그 파일, 자동화 스크립트와 조합해서 사용할 수 있다.

### 6. 컨텍스트 윈도우, 토큰 컨텍스트

- **컨텍스트 윈도우**는 클로드가 **한 번의 추론에서 참고할 수 있는 토큰의 총량**을 의미한다.
- 작업이 진행되면서 사용자 프롬프트, 클로드의 응답, 도구 호출 내용, 도구 호출 결과, 읽은 파일 내용, 명령어 실행 결과 등이 컨텍스트에 쌓인다.
- `CLAUDE.md` 기반 메모리도 컨텍스트에 포함되며, 엔터프라이즈 정책, 프로젝트 메모리, 사용자 메모리, 프로젝트별 로컬 메모리 등으로 나뉜다.
- 클로드 코드는 현재 작업 디렉터리에서 상위 디렉터리로 올라가며 관련 메모리를 탐색한다.
- 컨텍스트에 많은 정보를 무조건 넣는 것이 아닌, 작업 해결에 직접적으로 도움이 되는 **고신호 토큰**을 중심으로 컨텍스트를 구성하는 것이 중요하다.

> **이해 안갔던 부분**
>

책에서 가장 어려웠던 표현 중 하나가 “원하는 결과의 가능성을 최대화하는 최소한의 고신호 토큰 집합”이었다. “최소한의 고신호 토큰 집합”이라는 표현이 추상적으로 느껴졌던 거 같다.

정리해보면 고신호 토큰 집합이란, ‘작업 해결에 도움이 되는 꼭 필요한 정보의 집합’ 으로 이해했고, 작업에 필요한 정보를 선별해서 제공하는 것이 중요하다는 이야기 같다.

### 7. 컨텍스트 윈도우 관리 전략

클로드 코드는 긴 작업을 처리하기 위해 여러 방식으로 컨텍스트를 관리한다.

- **컴팩션**
    - 컨텍스트 사용량이 한계에 가까워지면 이전 대화를 요약해 컨텍스트를 정리하는 방식
    - 중요한 아키텍처 결정, 미해결 버그, 핵심 구현 내용은 남기고 중요도가 낮은 정보는 줄인다.
    - 이를 통해 긴 작업에서도 흐름을 이어갈 수 있다.
- **`/clear`**
    - 대화 기록을 초기화한 상태에서 새 작업을 시작하는 명령어
    - 이전 작업과 관련 없는 새로운 작업을 시작할 때 유용하다.
- **점진적 공개**
    - 모든 정보를 처음부터 컨텍스트에 넣지 않고, 필요한 순간에 관련 데이터를 불러오는 방식
    - 처음에는 파일 경로, 링크, 메타데이터처럼 가벼운 정보만 유지하고, 필요할 때 실제 내용을 읽는다.
    - Skills 시스템도 초기에는 메타데이터만 로드하고, 조건이 맞을 때 본문이나 참조 문서를 추가로 불러오는 방식으로 동작한다.
- **서브에이전트 활용**
    - 서브에이전트는 메인 에이전트와 별도의 컨텍스트 윈도우를 사용한다.
    - 대규모 코드베이스를 분석할 때 작업을 나누어 탐색하고, 메인 에이전트는 요약만 받아 종합할 수 있다.
    - 예를 들어 백엔드 구조, 프론트엔드 구조, 인증 흐름을 각각 다른 서브에이전트가 조사하도록 나눌 수 있다.
- **구조화된 메모 작성**
    - 체크리스트나 작업 기록을 마크다운 파일로 남겨 진행 상황을 관리하는 방식
    - 대규모 마이그레이션이나 여러 단계의 수정 작업에서 유용하다.

### 8. 클로드 코드의 다섯 가지 접근 영역과 핵심 기능

- 클로드 코드는 프로젝트 작업을 수행하기 위해 여러 영역에 접근한다.
- 주요 접근 영역은 **파일 시스템, 터미널, 웹, MCP 서버, 에디터 컨텍스트**로 나눌 수 있다.
- **파일 시스템**
    - `Read`, `Write`, `Edit`, `Glob`, `Grep` 도구를 통해 프로젝트 파일을 읽고, 검색하고, 수정할 수 있다.
    - 파일을 변경하는 작업은 사용자의 승인이 필요하며, 읽기 전용 작업은 승인 없이 수행될 수 있다.
- **터미널**
    - `Bash`를 통해 사용자의 셸 환경에서 명령어를 실행할 수 있다.
    - 로컬에 설치된 `pnpm`, `node`, `git` 같은 명령어와 환경 변수를 사용할 수 있다.
    - 테스트, 린터, 타입 체크처럼 작업 결과를 확인하는 과정에서 중요하게 활용된다.
- **웹**
    - `WebSearch`, `WebFetch`를 통해 인터넷에서 정보를 검색하거나 문서를 가져올 수 있다.
    - 라이브러리 문서 확인, 에러 메시지 검색, 최신 사용법 확인 등에 활용할 수 있다.
- **MCP 서버**
    - MCP를 통해 외부 도구와 데이터 소스에 연결할 수 있다.
    - 예를 들어 GitHub PR 확인, Sentry 오류 로그 분석, Figma 디자인 파일 확인 같은 작업이 가능하다.
    - MCP 서버는 로컬, 프로젝트 범위, 사용자 범위로 나누어 사용할 수 있다.
- **에디터 컨텍스트**
    - VS Code 확장이나 JetBrains 플러그인을 통해 IDE와 직접 연동할 수 있다.
    - 현재 열려 있는 파일, 선택한 코드, 에디터 상태 등을 작업 맥락으로 활용할 수 있다.

### 9. 클로드 코드의 4대 핵심 기능

- 클로드 코드는 자연어 요청을 바탕으로 코드베이스를 탐색하고, 문제를 분석하며, 필요한 작업을 수행할 수 있다.
- **자연어 기반 기능 구축**
    - 기능 요구사항을 자연어로 전달하면, 관련 코드를 탐색하고 필요한 변경을 수행한다.
- **심층 디버깅 및 문제 해결**
    - 버그 설명이나 오류 메시지를 바탕으로 원인을 분석하고 수정 방향을 제안한다.
    - 스택 트레이스를 활용하면 에러가 발생한 함수 호출 흐름을 추적할 수 있다.
- **코드베이스 탐색과 분석**
    - 새로운 프로젝트 구조를 파악하거나, 특정 기능이 어디에서 처리되는지 찾는 데 활용할 수 있다.
- **반복 태스크 자동화**
    - 반복되는 수정, 분석, 문서화 작업을 자동화할 수 있다.
    - 예) 린트 에러 수정, PR 설명 초안 생성, QA 체크리스트 생성

### 10. 대화형 모드와 비대화형 모드

클로드 코드는 작업 성격에 따라 **대화형 모드**와 **비대화형 모드**로 나누어 사용할 수 있다.

#### 대화형 모드

- 대화형 모드는 플래그 없이 `claude` 명령어만 입력해 실행하는 방식이다.
- REPL 스타일의 대화형 세션으로 동작한다.
    - **Read**: 입력을 읽고
    - **Eval**: 입력을 평가하고
    - **Print**: 결과를 출력하고
    - **Loop**: 다시 입력을 기다린다.
- 세션이 이어지기 때문에 질문을 주고받으며 탐색하거나, 중간에 계획을 확인하고 방향을 조정해야 하는 장기 작업에 적합하다.
- 대화형 모드에서는 슬래시 명령어를 사용해 설정, 커스텀 명령어, 에이전트, 스킬 같은 메타 작업도 수행할 수 있다.
- 프로젝트 설정은 보통 다음과 같은 구조로 관리할 수 있다.

    ```
    project-root/
    ├── .claude/
    │   ├── commands/          # 커스텀 슬래시 명령어
    │   ├── agents/            # 커스텀 에이전트 정의
    │   ├── skills/            # 에이전트 스킬 모듈
    │   ├── hooks/             # 이벤트 훅 스크립트
    │   ├── output-styles/     # 출력 스타일 템플릿
    │   └── settings.json      # 프로젝트별 클로드 코드 설정
    └── CLAUDE.md              # 프로젝트 지침서
    ```


#### 비대화형 모드

- 비대화형 모드는 `p` 또는 `-print` 플래그를 사용해 프롬프트를 바로 실행하는 방식

    ```bash
    git diff | claude -p "이 변경사항을 PR 설명으로 요약해줘"
    ```

- 프롬프트 결과만 출력한 뒤 종료되기 때문에 스크립팅이나 자동화 작업에 적합하다.

> **느낀점**
>

클코에 비대화형 모드가 있다는 사실을 처음 알았다..! 대화형 모드는 탐색하고 방향을 조정해야 하는 작업에 적합하고, 비대화형 모드는 결과가 명확한 반복 작업을 자동화할 때 적합하다고 이해했다.

특히 비대화형 모드는 이미 터미널에서 얻을 수 있는 출력값을 클로드 코드에 바로 전달할 수 있어서 설명과 같이 스크립팅이나 자동화 작업에 활용하기 좋아 보였다. 내일 당장 변경사항 요약 및 테스트 항목 정리, QA 체크리스트 자동화해보고 싶다.

### 11. 체크포인트와 되감기 시스템

- **체크포인팅**은 클로드 코드가 파일 편집 이력을 자동으로 추적하고, 원치 않는 변경을 이전 상태로 되돌릴 수 있게 해주는 **세션 단위의 안전망**이다.
- 클로드 코드는 프롬프트 단위로 체크포인트를 생성하며, 각 체크포인트는 작업 단계별로 돌아갈 수 있는 이력이 된다.
- 체크포인트 복구 옵션
    - 대화만 복구
    - 코드만 복구
    - 코드와 대화 모두 복구
- 체크포인트는 클로드 코드의 `Edit`, `Write` 같은 파일 편집 도구로 변경한 내용은 추적할 수 있지만, `Bash`를 통해 변경된 파일이나 외부에서 수정된 내용은 추적하지 못한다.
- 따라서 체크포인트는 작업 중간에 빠르게 되돌리기 위한 임시 안전망이고, 협업과 장기적인 변경 이력 관리는 Git을 기준으로 하는 것이 적절하다.
