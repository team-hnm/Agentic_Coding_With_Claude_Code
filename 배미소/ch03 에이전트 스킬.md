1. ‘클로드 스킬’ 학습

## 내용 정리

### 클로드 스킬의 개념
- 클로드의 기능을 확장하는 지식 모듈이다.
- `SKILL.md`(필수 지침 파일)와 선택적 지원 파일(스크립트, 템플릿 등)로 구성된다.
    - `SKILL.md`에서 상대 링크로 다른 파일을 참조하면, 클로드가 필요할 때만 그 파일을 로드한다.
- 프롬프트만 입력해도 클로드가 관련 스킬을 자동으로 찾아 쓰지만, `/<skill-name>`으로 직접 호출할 수도 있다.
- 저장 위치에 따라 적용 범위가 달라진다 (Enterprise / Personal / Project / Plugin). 마켓플레이스에서 플러그인 스킬을 설치할 수도 있다.

| **위치** | **경로** | **적용 대상** |
| --- | --- | --- |
| Enterprise | **관리 설정** 참조 | 조직의 모든 사용자 |
| Personal | **`~/.claude/skills/<skill-name>/SKILL.md`** | 모든 프로젝트 |
| Project | **`.claude/skills/<skill-name>/SKILL.md`** | 이 프로젝트만 |
| Plugin | **`<plugin>/skills/<skill-name>/SKILL.md`** | 플러그인이 활성화된 위치 |

### SKILL.md 파일 구조
- YAML 프론트매터 + 마크다운 본문 구조
- [🔗 프론트매터 필드](https://code.claude.com/docs/ko/skills#frontmatter-reference)
    - `name` 필드는 동명사형 (예: `analyzing-code-patterns`, `testing-react-components` )으로 무슨 행동을 하는 스킬인지 드러내야 한다.
    - `description` 필드에는 기능과 사용 트리거를 모두 포함해야 한다.
      > 클로드가 스킬을 찾을 때 해당 필드만 보고 판단하기 때문에
- `SKILL.md` 파일은 200줄 이내로 유지한다. 초과 시, 더 작은 단위로 분할하거나 `reference.md` 파일로 분리하여 단일 책임 원칙을 유지한다.

### 스킬의 세 단계 점진적 로딩
스킬은 컨텍스트 낭비를 막기 위해 필요한 만큼만 단계적으로 로드한다.
- **1단계 (메타데이터):** 스킬을 찾기 위한 필수 필드인 `name`과 `description` 만 항상 로드한다.
- **2단계 (지침):** 요청이 `description` 필드와 매칭되면 `SKILL.md` 전체를 컨텍스트 윈도우에 추가한다.
- **3단계 (지원 파일)**:  실제 필요할 때만 로드된다. 파일이 접근되기 전까지는 컨텍스트를 소비하지 않는다.
    - 마크다운 문서(`FORMS.md`, `REFERENCE.md` 등): 전문화된 가이드나 워크플로를 담는다.
    - 스크립트 파일: Bash로 실행되어 결정론적 작업을 수행하고, 코드 자체는 컨텍스트에 안 들어가고 실행 결과만 반영된다.
    - 참조자료: 데이터베이스 스키마, API 문서 등.

### 스킬 컨텍스트 비용 최적화
1. `descriptoin` 의 토큰 밀도를 높여 발견 가능성을 확보한다.
2. 클로드가 이미 알고 있는 범용 지식은 마크다운 본문에서 제거한다.
3. 지원 파일은 1단계 깊이만 유지한다. 중첩 구조로 참조 시, 부분만 로딩되어 정보가 누락될 수 있다.
4. 스킬의 입출력 예시 1~2개만 본문에 포함하고 나머지는 지원파일로 분리한다.
5. 결정론적 작업은 스크립트로 위임한다. (실행 결과만 컨텍스트에 반영하므로)
   > **💡 스크립트 우선 전략을 사용했을 때 효과적인 작업은 무엇일까?**
   >
   > 입력과 출력이 명확한 작업(특정 필드 값 추출, JSON/YAML 스키마 검증, 이미지 리사이즈, 엑셀 데이터 정제 등)

### 재사용 가능한 스킬 패턴
**특징**
1. 특정 프로젝트/기술에 종속되지 않는 일반적인 패턴을 다룬다.
2. `description`에 다양한 상황을 커버할 트리거 조건을 충분히 넣는다.
3. 체크리스트 형태의 명확한 단계를 가져야 다른 스킬과 조합하기 쉽다.

**예시**
- **프로젝트 특화 패턴**: 프로젝트 고유 아키텍처, 명명 규칙 등을 정의. 주로 Project 스킬로 저장해 팀원이 공유한다.
  > **💡 프로젝트 고유의 규칙을** **`CLAUDE.md`에 작성하는 것과 `SKILL.md`에 작성하는 것의 차이점이 뭘까? 스킬은 반복되는 작업을 위한 거라고 생각했는데, 정확히 어떤 부분을 `SKILL.md`에 정의해야 할지 궁금하다.**
  >
  > 핵심 차이는 ‘**항상 로드되는가, 필요할 때만 로드되는가’** 이다. `CLAUDE.md`는 세션 시작할 때 무조건 컨텍스트에 들어가야 할 **규칙**이고, `SKILL.md`는 트리거 조건에 매칭될 때만 로드되는 **절차**이다.
  >
  > 예를 들어, “Form 작성 시 항상 config 기반 선언형 빌더로 만든다"는 규칙은 `CLAUDE.md`에, "Form 필드 추가 시 Zod 스키마 → RHF 등록 → 유효성 메시지 연결한다”는 절차는 `SKILL.md`에 작성
- **도메인 특화 패턴**: 특정 기술 스택의 유의사항을 담는다.
  > ❗️ 이해가 안 되는 부분. 스킬 컨텍스트 비용 최적화 부분에서는 "클로드가 이미 알고 있는 범용 지식은 마크다운 본문에서 제거한다."라고 했는데, 특정 기술의 유의사항은 범용 지식이 아닐까? 싶다.
- **범용 유틸리티 패턴**: 코드 리뷰, 문서 생성, 테스트 작성, 리팩터링처럼 거의 모든 프로젝트에 필요한 작업. 재사용성이 가장 높아 Personal 스킬이나 마켓플레이스 공유에 적합하다.

### 스킬의 격리 실행 :context: fork 메커니즘
기본적으로 스킬은 부모 대화의 컨텍스트 안에서 실행되지만, 특정 상황에서는 격리 실행이 필요하다.
- `context: fork`를 프론트매터에 설정하면, 스킬 본문이 **서브에이전트를 구동하는 프롬프트**가 되어 부모 대화와 완전히 분리된 컨텍스트에서 실행된다.
- 서브에이전트는 부모 대화 히스토리에 접근할 수 없고, 작업이 끝나면 요약된 결과만 부모 대화로 반환된다.
- `agent` 필드는 `context: fork`가 설정된 경우에만 의미가 있으며, 서브에이전트의 동작 모드를 결정한다.
    - `Explore`: 코드베이스 탐색 최적화, 읽기 전용
    - `Plan`: 컨텍스트 수집 + 실행 계획 수립
    - `general-purpose`: 기본값, 제약 없는 범용 모드
- 적합한 상황: 대규모 코드베이스 분석, 세션 격리가 필요한 구조화된 워크플로(SPEC 문서 생성, TDD 사이클 등), 도구 전문화 기반 분석 작업.
- 적합하지 않은 상황: 부모 대화의 맥락을 참조해야 하는 스킬(서브에이전트는 히스토리를 상속받지 못하므로), 소규모 작업(서브에이전트 생성 오버헤드가 더 큼).
